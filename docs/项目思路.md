# 项目思路

## 1. 这个文档是做什么的

这是当前项目的文字版思维导图。

它的作用不是替代代码，而是帮助我们在正式开发前先理清：

- 这个项目要做什么
- 这个项目有哪些模块
- 这些模块之间是什么关系
- 我们应该按什么顺序一点点实现

## 2. 项目目标

项目名称：`battery-mes-platform`

项目定位：一个用于学习的微型动力电池 MES 云平台项目。

学习目标：

1. 学习前后端分离项目的基本搭建方式
2. 学习 FastAPI 和 Next.js 的协作方式
3. 学习数据库表设计和基础数据流转
4. 学习 MES / ERP / 工业互联网中常见的工单与追溯业务
5. 学习后续如何把 SQLite 升级到 PostgreSQL
6. 学习后续如何逐步接入 Redis、MQTT、Docker

1. 前端管理后台

9. 追溯查询

## 5. 核心业务流

后面做业务时，可以围绕下面这条主线来理解：

1. 创建工单
2. 生成或录入产品条码
3. 产品在工站扫码过站
4. 系统记录过站时间、工位、设备、人员
5. 设备上传模拟数据
6. 系统判断数据是否异常
7. 出现异常时记录报警
8. 最终通过条码查询完整追溯链路

## 6. 每个模块在真实 MES 中对应什么

### 工单管理

对应生产任务管理。比如今天要生产多少电芯、多少模组、多少电池包。

### 条码管理

对应产品唯一身份标识。真实工厂里，一个产品通常会有唯一条码或序列号。

### 过站记录

对应产品在每一道工序上的流转记录。系统需要知道它什么时候到过哪个工位。

### 设备数据模拟

对应产线设备、测试机、采集器上传的数据。学习阶段我们先用程序模拟。

### 异常报警

对应生产过程中的超限、漏检、失败、设备异常等情况。

### 追溯查询

对应 MES 的核心价值之一。出现问题时，可以按条码反查工单、工位、设备、参数、报警。

## 8. 当前我们要坚持的开发原则

1. 每次只做一个小功能
2. 每次只增加少量依赖
3. 每次改动前先说明改哪些文件
4. 每次写完都要能运行、能测试
5. 尽量写清楚中文注释
6. 不随意重构无关内容

## 9. 你现在适合顺手了解的概念

如果你觉得有些词还比较陌生，可以先建立一个模糊印象，后面边做边补：

- `Git`：版本管理工具
- `GitHub`：远程代码托管平台
- `Repository`（仓库）：一个项目在 Git / GitHub 中的存放单位
- `Commit`：一次版本快照
- `Branch`：分支
- `API`：前后端之间的数据接口
- `ORM`：用代码操作数据库的一种方式

- `追溯`：按产品条码回查生产全过程

## 10. 当前接口

### 健康检查

- 请求方式：`GET`
- 请求地址：`/health`
- 返回示例：`{"status": "ok"}`
- 业务作用：让开发人员或监控系统确认后端服务是否正在正常运行

### 工单列表

- 请求方式：`GET`
- 请求地址：`/work-orders`
- 返回示例：`[{"id": 1001, "product_name": "动力电池模组", "planned_quantity": 500, "status": "生产中"}]`
- 业务作用：查询当前系统中的生产任务列表，是 MES 工单管理的入口

### 工单详情

- 请求方式：`GET`
- 请求地址：`/work-orders/{work_order_id}`
- 返回示例：`{"id": 1001, "product_name": "动力电池模组", "planned_quantity": 500, "status": "生产中"}`
- 业务作用：根据工单编号查询单个工单的完整信息
- 异常情况：如果工单编号不存在，返回 `404`

### 创建工单

- 请求方式：`POST`
- 请求地址：`/work-orders`
- 请求示例：`{"product_name": "电芯测试工单", "planned_quantity": 200, "status": "待生产"}`
- 返回示例：`{"id": 1004, "product_name": "电芯测试工单", "planned_quantity": 200, "status": "待生产"}`
- 业务作用：从前端接收新工单数据，由后端生成工单编号并保存到当前模拟工单列表

## 11. 当前前后端工单交互思路

### 工单列表展示

```text
页面打开
→ 前端 useEffect 执行一次
→ fetch 请求 GET /work-orders
→ 后端返回工单数组
→ 前端 setWorkOrders 保存数组
→ 页面用 map 把每个工单渲染成一张卡片
```

### 工单详情展示

```text
用户点击某张工单卡片
→ 前端拿到当前工单 id
→ 调用 loadWorkOrderDetail(id)
→ fetch 请求 GET /work-orders/{id}
→ 后端返回单个工单详情
→ 前端 setSelectedWorkOrder 保存当前工单
→ 右侧详情区域重新渲染
```

### 创建工单

```text
用户填写产品名称、计划数量、状态
→ 前端表单提交
→ fetch 发送 POST /work-orders
→ body 中携带 JSON 工单数据
→ 后端接收请求体并生成新 id
→ 后端把新工单加入 work_orders
→ 前端重新请求工单列表
→ 页面显示新工单并选中它
```

### 当前学习重点

```text
前端不是直接拥有真实业务数据。
前端通过接口向后端请求数据。
后端负责提供数据结构和业务结果。
前端负责保存状态、响应用户操作、展示结果。
```

## 13. 自动测试

当前自动测试位于 `backend/tests/test_main.py`。

测试覆盖：

- `GET /health` 返回状态码 `200`
- `GET /work-orders/1001` 返回完整工单信息
- `GET /work-orders/9999` 因工单不存在而返回状态码 `404`
- `GET /work-orders/abc` 因工单编号不是整数而返回状态码 `422`

运行测试：

```powershell
$env:PYTHONPATH='backend'
.\backend\.venv\Scripts\python.exe -m unittest discover -s backend\tests -v
```
