# 项目思路

## 1. 这个文档是做什么的

这是当前项目的文字版思维导图。

它的作用不是替代代码，而是帮助我们在正式开发前先理清：

- 这个项目要做什么
- 这个项目有哪些模块
- 这些模块之间是什么关系
- 我们应该按什么顺序一点点实现

## 2. 项目目标

项目名称：`battery-mes-platform`

项目定位：一个用于学习的微型动力电池 MES 云平台项目。

学习目标：

1. 学习前后端分离项目的基本搭建方式
2. 学习 FastAPI 和 Next.js 的协作方式
3. 学习数据库表设计和基础数据流转
4. 学习 MES / ERP / 工业互联网中常见的工单与追溯业务
5. 学习后续如何把 SQLite 升级到 PostgreSQL
6. 学习后续如何逐步接入 Redis、MQTT、Docker

## 4. 系统模块

先把项目拆成几个容易理解的小模块：

1. 前端管理后台
2. 后端 API 服务
3. 数据库
4. 工单管理
5. 条码管理
6. 条码过站记录
7. 设备数据模拟
8. 异常报警
9. 追溯查询

## 5. 核心业务流

后面做业务时，可以围绕下面这条主线来理解：

1. 创建工单
2. 生成或录入产品条码
3. 产品在工站扫码过站
4. 系统记录过站时间、工位、设备、人员
5. 设备上传模拟数据
6. 系统判断数据是否异常
7. 出现异常时记录报警
8. 最终通过条码查询完整追溯链路

## 6. 每个模块在真实 MES 中对应什么

### 工单管理

对应生产任务管理。比如今天要生产多少电芯、多少模组、多少电池包。

### 条码管理

对应产品唯一身份标识。真实工厂里，一个产品通常会有唯一条码或序列号。

### 过站记录

对应产品在每一道工序上的流转记录。系统需要知道它什么时候到过哪个工位。

### 设备数据模拟

对应产线设备、测试机、采集器上传的数据。学习阶段我们先用程序模拟。

### 异常报警

对应生产过程中的超限、漏检、失败、设备异常等情况。

### 追溯查询

对应 MES 的核心价值之一。出现问题时，可以按条码反查工单、工位、设备、参数、报警。

## 7. 分阶段计划

### 第一阶段：项目初始化

- 创建目录结构
- 初始化 Git
- 初始化 GitHub 仓库关联
- 初始化前端
- 初始化后端

### 第二阶段：前后端最小跑通

- 前端能打开页面
- 后端能提供一个测试接口
- 前端能调用后端接口

### 第三阶段：工单基础功能

- 创建工单
- 工单列表
- 工单详情

### 第四阶段：条码与过站

- 条码建档
- 扫码过站
- 查询过站记录

### 第五阶段：设备数据与报警

- 模拟设备上传数据
- 设定阈值
- 异常记录与报警展示

### 第六阶段：追溯查询

- 按条码查询完整生产过程
- 关联工单、过站、设备数据、报警信息

## 8. 当前我们要坚持的开发原则

1. 每次只做一个小功能
2. 每次只增加少量依赖
3. 每次改动前先说明改哪些文件
4. 每次写完都要能运行、能测试
5. 尽量写清楚中文注释
6. 不随意重构无关内容

## 9. 你现在适合顺手了解的概念

如果你觉得有些词还比较陌生，可以先建立一个模糊印象，后面边做边补：

- `Git`：版本管理工具
- `GitHub`：远程代码托管平台
- `Repository`（仓库）：一个项目在 Git / GitHub 中的存放单位
- `Commit`：一次版本快照
- `Branch`：分支
- `API`：前后端之间的数据接口
- `ORM`：用代码操作数据库的一种方式
- `MES`：制造执行系统
- `追溯`：按产品条码回查生产全过程
